2  模块导论与考核说明

2.1 为什么金融学生需要学习文本智能

金融市场中大量重要信息并不以整洁表格出现，而是写在公告、年报、问询函回复、投资者关系活动记录和监管文件中。投资者、分析师、监管者和企业管理者都需要从这些文本中提取事实、识别风险、比较公司、追踪事件进展。

本模块关注的不是“让大模型写一段总结”，而是把公告中的关键信息变成可校验、可比较、可复用的结构化结果。例如：

  • 从减持公告中抽取股东名称、减持比例、减持期间和减持原因；
  • 从年报风险因素中抽取风险类别、风险描述和管理层应对；
  • 从问询函与回复中判断监管问题是否被公司逐项回应；
  • 从回购方案、进展和结果公告中构建事件时间线。

一个可信的金融文本智能系统应当能回答三个问题：

  1. 结果从哪里来？
  2. 字段是否符合事先定义？
  3. 错了以后怎么发现和改进？

2.2 为什么使用巨潮公告

巨潮资讯网是上市公司公开披露信息的重要入口。它适合本课程项目，原因有三点：

  • 公开性：项目主数据来自公开可访问的信息披露材料。
  • 金融含义明确：公告类型对应真实金融问题，如治理、风险、分红、回购、诉讼、业绩预告等。
  • 工程挑战适中：PDF、长文本、表格、标题层级和证据回填都能训练学生完整处理真实任务。

所有小组的项目主数据必须来自巨潮资讯网。教师给出的选题池只是参考，学生可以自拟题目，但必须证明巨潮上能找到足够样本。

2.3 为什么不训练大模型

本课程对象是金融学本科三年级学生。后半程目标不是训练一个大模型，而是学会把已有工具组合成一个可复现的金融文本智能项目。

项目技术链条包括：

巨潮公告数据
  -> PDF 文档解析
  -> 章节定位与检查
  -> LLM API 调用
  -> Pydantic Schema 校验
  -> 工作流串联
  -> 人工评估与优化

这条链路更接近金融机构和研究团队在真实项目中的工作方式：先定义问题和字段，再用 API 与工作流解决具体任务。

2.4 本课程中的 Vibe Coding

本课程采用受控 Vibe Coding。AI coding agent 是开发助理，不是项目负责人。

学生负责：

  • 定义金融问题；
  • 判断公告数据是否匹配；
  • 设计字段、schema、验收标准；
  • 运行代码、检查输出、记录错误；
  • 解释项目方法与结果。

AI coding agent 可以帮助：

  • 生成项目骨架；
  • 写小模块代码；
  • 解释报错；
  • 补充日志与测试；
  • 重构重复代码；
  • 帮助整理 README 与展示材料。

受控 vibe coding 的基本流程是：

Spec -> Plan -> Generate -> Run -> Repair -> Review

也就是先写清规格，再让 AI 提供计划或代码；学生必须本地运行、修复报错、检查结果，并在 ai_worklog.md 中记录过程。

2.6 与课程总评的关系

若课程总评采用以下结构：

考勤 10%
项目一：结构化建模 20%
项目二：金融文本分析 20%
期末考试 50%

则本讲义主要负责：

  • 项目二：金融文本智能项目；
  • 期末考试中模块 B 相关内容。

项目二内部采用 100 分制，最后折算到课程总评中的项目二部分。期末考试覆盖全课程，模块 B 可能以概念题、简答题、案例分析题形式出现。

2.7 项目二评分公式

项目二内部得分建议使用：

项目二内部得分 = min(100, 基础质量分 + 挑战分 × 难度系数)

其中：

  • 基础质量分：80 分；
  • 挑战分：20 分；
  • 难度系数：0.9 / 1.0 / 1.1。

不建议把总分全部乘以难度系数。这样可以鼓励挑战，也避免基础组选题做得扎实的同学被过度惩罚。

2.9 挑战分 Rubric

维度 分值 说明
数据规模与质量 8 达到所选档位的数据量，去重、失败记录、样本覆盖合理
任务复杂度 8 多字段、多文档匹配、时间线、闭环、评分、复杂 section 等

2.10 难度档位

档位 系数 参考数据量 典型任务
基础档 0.9 30–60 份 PDF 或公告记录 单公告类型，4–6 个字段，不要求跨公告匹配，顺序工作流即可
标准档 1.0 80–150 份 PDF，或 30–50 组事件 单一复杂公告类型或两个相关公告类型，6–10 个字段，有 section 检查、证据回填和评估
挑战档 1.1 150+ 份 PDF，或 50+ 组多文档匹配事件 多公告类型、多阶段事件、闭环分析、时间线、评分，10 个以上字段

数据量不是唯一标准。若选题需要多公告匹配、跨期追踪、复杂表格解析或人工标注评估，即使 PDF 数量不特别大，也可申请挑战档。

2.10.1 基础档保底路径

对只有 Python 基础、第一次使用 AI coding agent 的小组，建议先完成以下保底路径：

单一公告类型
  -> 30 份左右公告或一批可检查样本
  -> 4–6 个核心字段
  -> metadata + PDF/解析文本可追溯
  -> section 检查
  -> Pydantic 校验
  -> 10 条左右人工评估

基础档不追求复杂多公告匹配，也不要求把所有环节都自动化到完美。只要能证明数据真实、字段清楚、结果有证据、错误有记录，就可以形成合格项目。

2.11 选题通过标准

学生自拟题目需要通过以下检查：

维度 通过标准
巨潮数据可获得 能在巨潮找到足够样本
金融问题清楚 有风险、合规、治理、业绩、分红、市场沟通等金融含义
字段可结构化 能写成 JSON / CSV，能判断对错
证据可追溯 能提供公告标题、日期、页码、原文片段
6 周可完成 工作量合理，不依赖不可控数据
难度可声明 能说明为何是 0.9 / 1.0 / 1.1

2.13 期末考试范围

期末考试覆盖全课程，模块 B 相关内容约占期末考试 30%，具体以正式试卷为准。

模块 B 不考具体工具安装命令，也不要求背诵 API 代码。重点考察学生是否理解金融文本智能项目的关键概念、流程设计和错误分析。案例分析题会与同学们做过的项目类型相关。

部分 预计题量 模块 B 占期末比例 考查重点
客观题 2–4 小题 约 8%–10% 基本概念、术语辨析、流程顺序
简答题 1 小题 约 8%–10% 为什么这样设计、如何避免常见错误
案例分析题 1 小题 约 10%–12% 结合金融公告项目，分析数据、schema、section、workflow 或评估问题

模块 B 考点包括：Vibe coding、AI coding agent、巨潮数据与 metadata、LLM API、Prompt、Pydantic Schema、MinerU、Section Routing / Checking、Evidence、Workflow、Human-in-the-loop、Evaluation metrics、Hallucination、Null rule、Skills / Rules。

2.14 合规与安全

本项目只面向公开可访问数据。必须遵守：

  • 不绕过登录、验证码或访问限制；
  • 抓取脚本必须限速，并保留日志；
  • API Key 必须放入 .env，不得提交到 Git 或作业平台；
  • 不把模型输出当作未经验证的事实；
  • 每个关键字段都应能回到公告原文或明确标注 null；
  • 使用 AI coding agent 的过程必须保留 ai_worklog.md 或等价记录。
