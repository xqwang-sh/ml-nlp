10  Week 15：评估、提示词优化、Skills 与展示工程化

10.1 本周目标

Week 15 的任务是把“系统能跑”推进到“结果可信、过程可复现、错误可解释”。

完成本周后，学生应该能够：

  1. 设计轻量人工评估；
  2. 识别错误来源；
  3. 优化 prompt、schema、section rules 和 workflow；
  4. 编写项目 rules / skills，让 AI coding agent 更稳定地帮助项目；
  5. 整理 final repo、demo 和 slides。

10.2 评估框架

金融文本智能项目不能只展示成功样例。评估要回答：系统在多少样本上可靠，错误主要来自哪里，下一步应该改什么。

本课程建议五类指标：

指标 问题 示例
Data Quality 抓到的数据是否真的属于目标任务？ 标题匹配率、PDF 可打开率、重复率
Section Quality 目标章节是否找对？ 关键章节找到率、页码正确率、表格损坏率
Extraction Quality 字段是否抽对？ 字段准确率、缺失率、格式错误率
Evidence Quality 证据是否支持字段？ evidence 命中率、页码正确率
Pipeline Stability 工作流是否稳定？ 批处理成功率、失败重试率、平均耗时

10.2.1 轻量人工评估

每组抽样人工标注：

难度 人工标注建议
基础 10–15 份 PDF
标准 15–25 份 PDF
挑战 25+ 份 PDF 或 10+ 对匹配样本

评估表字段：

doc_id,field_name,predicted_value,gold_value,is_correct,evidence_correct,error_type,notes

error_type 建议使用固定枚举：

  • data_error
  • parse_error
  • section_error
  • prompt_error
  • schema_error
  • hallucination
  • normalization_error
  • workflow_error
  • human_label_unclear

10.3 错误驱动优化

优化不是盲目换模型，也不是只改 prompt。先判断错误来自哪里。

错误类型 表现 可能原因 修复方法
抓取错误 标题不相关，PDF 缺失 关键词过宽、下载失败未记录 收窄关键词、补充失败日志、人工抽查
PDF 解析错误 表格丢列、页码错乱 PDF 版式复杂，解析结果未检查 换解析策略、保留原 PDF、标记人工复核
Section 错误 命中目录或无关章节 include 过宽，exclude 不足 加排除词、页码扩展、最小长度规则
Prompt 错误 字段解释不清，输出不稳定 字段定义模糊，没有 null rule 重写字段说明，加 JSON 示例
Schema 错误 Pydantic 大量失败 类型过窄或字段设计不合理 拆分原文值和标准化值，调整枚举
模型幻觉 evidence 不在输入中 没有强制证据规则 要求 evidence 原文匹配，失败则 null
工作流失败 单条能跑，批量失败 路径、缓存、异常处理混乱 加统一入口、日志、重试和断点
展示不可复现 Demo 依赖临时文件 README 不完整，命令不清 固化配置、准备 sample outputs

10.4 Prompt 优化

Prompt 优化应围绕具体错误进行。

有效做法：

  • 字段定义更具体；
  • 给出枚举值；
  • 日期和金额格式统一；
  • 不确定输出 null；
  • evidence 必须来自输入文本；
  • few-shot 示例只放 1–2 个，避免过长；
  • 对失败案例写反例提示；
  • 将长任务拆成分类、抽取、验证三个小任务。

10.4.1 抽取 Prompt 模板

你是金融公告结构化抽取助手。

任务：
请只根据输入文本抽取字段，输出合法 JSON。

规则：
1. 不得根据常识补全。
2. 不确定或文本中不存在时输出 null。
3. 每个关键字段必须提供 evidence_text。
4. evidence_text 必须是输入文本中的原文片段。
5. 日期使用 YYYY-MM-DD；无法标准化则保留原文并在标准化字段输出 null。
6. 金额先保留原文表达，不要自行换算单位，除非文本明确。

输入文本：
{section_text}

字段说明：
{schema_description}

10.4.2 JSON 修复 Prompt

当模型输出无法通过 Pydantic 校验时，可以让模型只修复格式：

下面是模型输出和 Pydantic 校验错误。
请只修复 JSON，使其符合 schema。
不得添加输入文本中没有的信息。

原始输出：
{raw_output}

校验错误：
{validation_error}

Schema：
{json_schema}

10.5 Schema 优化

常见优化方式：

  • 将模糊字段拆分为原文值和标准化值；
  • 对枚举字段使用 Literal；
  • 对证据使用嵌套模型；
  • 将多主体字段设计为 list；
  • 保存 validation error，用于下一轮 prompt 改进。

示例：

class MoneyValue(BaseModel):
    raw_text: Optional[str] = None
    amount: Optional[float] = None
    unit: Optional[Literal["元", "万元", "亿元"]] = None
    evidence_text: Optional[str] = None

这样比只写一个 amount: float 更稳健，因为公告中金额单位经常变化。

10.6 Section 优化

Section 优化不是追求自动化到完全不看，而是减少明显误定位。

可用策略：

  • 增加关键词和同义词；
  • 增加排除词，如“目录”“释义”“声明”；
  • 设置最小内容长度；
  • 对跨页章节设置页码扩展；
  • 对表格损坏建立手工兜底逻辑；
  • 将 section preview 输出给人工快速检查。

每次修改 section rules 后，应重新跑一小批样本，并更新 section_check_report.md。

10.7 Skills / Rules

这里的 skills 不是给最终 LLM 抽取字段的 prompt，而是给“帮你写代码的 AI coding agent”的长期项目说明。不同工具叫法可能不同：TRAE 可使用 Rules & Skills；Claude Code / Kimi Code 有类似能力；Codex 可用 AGENTS.md 存放项目说明和工作规则。

它们的作用是减少重复解释，让 AI 每次写代码时都遵守项目边界。

10.7.1 层次一：项目规则文件

AGENTS.md、CODING_RULES.md 或 CLAUDE.md 建议包含：

# Agent Instructions for CNINFO Project

## Project Goal
本项目从巨潮资讯网公告 PDF 中抽取结构化金融事件信息。

## Data Rules
- 主数据必须来自巨潮资讯网公开公告。
- 不绕过登录、验证码或访问限制。
- 原始 PDF 不得修改。

## Secret Rules
- 不要把 API Key 写入代码。
- 不要提交 .env。
- 只维护 .env.example。

## Coding Rules
- 每个脚本必须支持命令行运行。
- 所有输出写入 outputs/。
- 所有错误写入 outputs/logs/。
- 修改代码后必须说明运行哪个命令验证。

## Extraction Rules
- 字段值必须来自公告文本。
- 每个关键字段要有 evidence_text。
- 无法判断时输出 null。
- 抽取结果必须通过 Pydantic 校验。

10.7.2 层次二：任务 Prompt 模板

例如：

  • 帮我改抓取脚本；
  • 帮我修 JSON validation 错误；
  • 帮我写 section checker；
  • 帮我审查 README 是否可复现。

每个任务 prompt 都要说明输入、输出、约束和验收命令。

10.7.3 层次三：可复用 Skill

如果工具支持 skill，可以建立：

skills/cninfo-crawler/SKILL.md
skills/mineru-parser/SKILL.md
skills/section-checker/SKILL.md
skills/pydantic-extractor/SKILL.md
skills/evaluator/SKILL.md

每个 skill 包含：

  • 什么时候使用；
  • 输入文件；
  • 输出文件；
  • 不允许做什么；
  • 验收命令。

10.9 课堂练习

  1. 抽样人工评估 5 条结果。
  2. 将错误按类型分类。
  3. 根据错误修改 prompt、schema 或 section rules。
  4. 编写 AGENTS.md 或 CODING_RULES.md。
  5. 让 AI coding agent 作为 reviewer 检查最终 repo。
  6. 准备 3 分钟 demo 脚本。

10.11 常见错误

  • 只展示成功案例；
  • 没有人工评估；
  • 优化只改 prompt，不查 section 和数据；
  • AI 使用过程不可追踪；
  • README 缺少复现命令；
  • Demo 依赖临时文件或本地绝对路径；
  • evaluation 只写“效果不错”，没有样本和错误类型；
  • skills/rules 写成空泛口号，没有项目约束。

10.12 教师提示

本周教师可以要求每组展示一个失败案例。好的项目不是没有失败，而是能说清：

  1. 失败在哪里；
  2. 为什么失败；
  3. 如何发现；
  4. 如何修复；
  5. 修复后是否真的改善。
