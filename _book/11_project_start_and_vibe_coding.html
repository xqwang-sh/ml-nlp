3  Week 11：项目启动、巨潮数据观察与受控 Vibe Coding 入门

3.1 本周目标

Week 11 的核心是先看数据、定问题、装工具、建骨架。

完成本周后，学生应该能够：

  1. 在巨潮资讯网上观察不同公告类型；
  2. 从公告类型出发提出一个金融文本分析选题；
  3. 理解教师给出的选题只是参考，能够自拟题目；
  4. 安装 TRAE，或确认 Claude Code / Codex / Kimi Code 等工具可用；
  5. 使用 AI coding agent 建立最小项目骨架；
  6. 写出 topic_proposal.md 和 cninfo_data_discovery.md。

3.2 为什么本周先看数据

很多项目失败，不是因为模型不够强，而是因为一开始没有弄清楚数据长什么样。

例如，“分析年报风险”听起来很合理，但如果没有进一步限定行业、年份、风险章节、字段和输出形式，就会变成一个过大的题目。相反，“抽取新能源行业 2022–2025 年年报风险因素中的供应链风险、价格风险和政策风险，并比较风险表述变化”就更容易落地。

本周的基本原则是：

公告类型 -> 金融问题 -> 目标字段 -> 结构化输出 -> 难度档位

3.3 11.1 巨潮数据地图与选题发现

3.3.1 巨潮在本项目中的角色

巨潮资讯网提供上市公司公告查询和 PDF 下载入口。本项目的主数据必须来自巨潮公告。其他资料可以作为背景知识，但不能替代项目主数据。

学生需要观察的公告类型包括：

  • 定期报告：年报、半年报、季报；
  • 临时公告：回购、增减持、重大合同、诉讼仲裁、关联交易、担保、分红、股权激励；
  • 监管互动：问询函、关注函、回复公告；
  • 投资者关系活动记录。

3.3.2 从公告类型到项目问题

一个好选题不是“做某类公告总结”，而是回答一个具体金融问题。

示例：

观察：很多公司披露“关于股东减持计划的公告”。
金融问题：大股东减持是否可能形成市场风险信号？
目标字段：股东名称、减持数量、减持比例、减持期间、减持原因、是否为控股股东。
输出形式：股东减持事件台账 + 风险提示卡。
难度：基础档 0.9。

再看一个更复杂的例子：

观察：上市公司会先披露回购方案，之后披露进展公告和结果公告。
金融问题：公司回购计划是否按承诺执行？回购金额与上限差距多大？
目标字段：方案日期、计划金额区间、计划股份数量、进展金额、最终完成金额、完成比例。
输出形式：回购生命周期时间线 + 完成度统计。
难度：挑战档 1.1。

3.4 11.2 Vibe Coding 工具安装与工作流

3.4.1 工具选择

默认推荐使用 TRAE。已经拥有 Claude Code、Codex、Kimi Code 或其他 AI coding agent 的学生可以继续使用原工具。

课程不以工具品牌评分。评分只看：

  • 项目是否可复现；
  • AI 使用过程是否有记录；
  • 学生是否能解释关键代码；
  • 结果是否有证据和评估。

3.4.3 受控 Vibe Coding 六步法

Spec -> Plan -> Generate -> Run -> Repair -> Review
步骤 学生要做什么
Spec 写清任务、输入、输出、约束和验收标准
Plan 让 AI 先给实现计划，不急着写代码
Generate 让 AI 只生成一个小模块
Run 学生在本地运行命令
Repair 把完整报错贴回 AI，让它只修改必要部分
Review 检查输出、记录过程、解释关键代码

3.4.4 为什么不能让 AI 一次性写完整项目

一条 prompt 让 AI “帮我做一个巨潮公告分析项目”，通常会导致：

  • 文件结构看起来完整，但无法运行；
  • 字段定义不稳定；
  • API Key 处理不安全；
  • 数据来源和合规边界不清；
  • 学生无法解释代码；
  • 错误难以定位。

因此本课程要求把任务拆成小模块。每次只让 AI 处理一个可验证的步骤。

3.4.5 Prompt 对比

坏 prompt：

帮我做一个巨潮公告分析项目。

好 prompt：

请只帮我生成一个 run.py，使它支持 python run.py --help，并显示本项目可用命令。
不要生成其他文件。请给出运行命令和最小测试。

更完整的开发 prompt：

你是我的 Python 项目助教。请帮我创建一个最小可运行的课程项目骨架，
但不要实现业务逻辑。

背景：金融文本智能课程，数据来自巨潮资讯网 PDF。
要求：
1. 创建标准目录：data、src、configs、outputs、tests。
2. 写一个 run.py，支持 python run.py --help。
3. 写 README.md，说明项目目标、目录结构和运行方法。
4. 写 .env.example，但不要出现真实 API Key。
5. 写 ai_worklog.md 模板。
6. 不要抓取数据，不要调用 LLM，不要写复杂功能。

3.5 11.3 最小项目骨架

本周建议项目结构如下：

project_name/
  README.md
  ai_worklog.md
  .env.example
  requirements.txt
  configs/
    task.yaml
  data/
    metadata/
    pdf/
    parsed/
  src/
    __init__.py
    cninfo/
    llm/
    mineru/
    workflow/
    evaluation/
  outputs/
    logs/
    results/
    reports/
  tests/

本周不要求抓取大量数据。项目骨架的验收标准是：

  • python run.py --help 可以运行；
  • README 能说明项目目标和目录；
  • .env.example 只有变量名和占位符；
  • 有 ai_worklog.md 或周记录模板；
  • 不包含真实 API Key。

如果学生第一次使用 AI coding agent，可以先运行讲义目录中的离线 starter：

cd starter_example
python3 -m pip install -r requirements.txt
python3 run.py all

starter 的作用是帮助学生看懂项目链路，不是最终答案。正式项目仍需替换为本组选题、真实 metadata、真实 PDF 或解析文本。

3.6 11.4 选题可行性工作坊

每组需要把“看到的数据”转化为可执行项目。选题四问：

  1. 数据来自巨潮哪类公告？
  2. 字段是否能结构化？
  3. 证据是否能追溯？
  4. 六周是否做得完？

3.6.1 绿灯、黄灯、红灯

状态 判断标准 教师反馈
绿灯 数据可获得，字段清楚，金融问题明确，工作量合适 可以进入 Week 12 抓取
黄灯 方向有价值，但字段、数据量或范围需要收缩 修改后再确认
红灯 数据不是巨潮来源，题目过大或字段无法判断对错 换题或大幅重做

3.9 常见错误

  • 选题太大，例如“分析所有上市公司年报风险”；
  • 选题太空，例如“总结公司公告”；
  • 数据不是巨潮来源；
  • 字段无法判断对错；
  • 没有证据页码或原文片段；
  • 过早写代码，没搞清公告类型；
  • 让 AI 一次性生成整个项目；
  • API Key 出现在 README、截图或代码里。

3.10 教师提示

本周教师应重点追问三个问题：

  1. 你们为什么选择这个公告类型？
  2. 你们最终输出的表格或 JSON 长什么样？
  3. 哪些字段最容易判断错？

如果学生回答不出第三个问题，说明他们还没有真正理解任务边界，应先缩小选题。
