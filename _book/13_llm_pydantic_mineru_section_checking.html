6  Week 13：MinerU 解析、Section 检查、Schema 与字段抽取

6.1 本周目标

Week 13 的任务是把 PDF 变成可处理文本，并把“想抽取的字段”变成可验证的结构化输出。

完成本周后，学生应该能够：

  1. 用 MinerU API 把公告 PDF 转为 Markdown；
  2. 对 Markdown 解析结果做抽样检查；
  3. 用 section rules 找到目标文本片段；
  4. 用 Pydantic 写项目字段 Schema；
  5. 用规则 baseline 或硅基流动 API 抽取字段；
  6. 对自动识别的 section 做人工检查和规则修正；
  7. 输出带证据的结构化抽取结果。

6.2 本周主线

PDF
  -> MinerU 解析
  -> Markdown / JSON
  -> Section Routing
  -> Section Checking
  -> 规则 baseline 或硅基流动 LLM 抽取
  -> Pydantic 校验
  -> 带 evidence 的结果

6.3 MinerU API：PDF 转 Markdown

本周先讲 MinerU，因为后面的 section routing、字段抽取和 evidence 都依赖解析文本。如果 PDF 没有被可靠地转成 Markdown，后面的 LLM 抽取得再漂亮也不可信。

MinerU 的作用是把 PDF 转为 Markdown、JSON 或其他更容易处理的文本格式。与普通 PDF 文本提取相比，公告 PDF 常见的难点包括：

  • 表格；
  • 页眉页脚；
  • 多栏排版；
  • 目录；
  • 标题层级；
  • 跨页段落；
  • 扫描件或复杂版式。

课堂 demo 的标准路径是：

data/pdf/*.pdf
  -> MinerU API
  -> data/parsed/markdown/*.md
  -> data/parsed/parsed_docs.jsonl

parsed_docs.jsonl 是后续脚本的统一输入。即使不同小组使用不同解析工具，最终也应该整理成相同结构：

{
  "doc_id": "1223877447",
  "stock_code": "600033",
  "stock_name": "福建高速",
  "title": "福建高速2024年年度报告",
  "pdf_path": "data/pdf/600033_1223877447.pdf",
  "markdown_path": "data/parsed/markdown/1223877447.md",
  "parser": "mineru",
  "pages": [
    {"page_no": 1, "text": "Markdown or parsed text here"}
  ]
}

6.3.1 .env.example

.env.example 只写变量名和占位符，不写真实 Key：

LLM_PROVIDER=siliconflow
LLM_BASE_URL=https://api.siliconflow.cn/v1
LLM_API_KEY=your_key_here
LLM_MODEL=Qwen/Qwen3-8B

MINERU_API_KEY=your_mineru_key_here

真实 .env 不得提交到 Git 或作业平台。

6.3.2 MinerU 解析检查

运行课堂真实 demo：

cd labs/cninfo_annual_report_risk_lab
python3 -m pip install -r requirements.txt
cp .env.example .env
# 在 .env 中填入真实 MINERU_API_KEY
python3 src/pipeline_run.py --step parse

正式项目和课堂 demo 都要求用 MinerU API 解析 PDF。没有真实 Key、API 返回失败、或解析结果为空时，脚本应直接退出，不允许静默使用本地解析替代。

6.4 Pydantic Schema 与结构化抽取

6.4.1 为什么先定义 Schema

Pydantic Schema 是字段契约。它能告诉模型、代码和评分标准：结果应该长什么样。

先定义 Schema 的好处：

  • 明确字段；
  • 限制类型；
  • 便于验证；
  • 便于评分；
  • 便于根据错误修复 prompt；
  • 避免“这次字段叫 amount，下次叫 money”的混乱。

6.4.2 常用字段类型

类型 用法
str 公司名、公告标题、原文金额
int 页码、次数、年限
float 标准化金额、比例
Optional 可能缺失的字段
Literal 枚举值，如事件类型
list 多个主体、多个问题、多个证据
嵌套模型 证据、金额、日期范围等复杂对象

6.4.3 Null Rule

不确定时输出 null，不能编造。

这是金融公告抽取中非常重要的规则。模型如果根据常识补全字段，结果看起来完整，但实际不可用。

6.4.4 示例：沪主板年报风险披露 Schema

from typing import Optional, Literal
from pydantic import BaseModel, Field


class Evidence(BaseModel):
    text: str = Field(description="支持风险类别判断的年报原文")
    page_no: Optional[int] = Field(default=None, description="证据所在页码")


class RiskCategory(BaseModel):
    category: Literal["市场风险", "行业竞争风险", "经营风险", "财务风险", "政策与合规风险", "其他风险"]
    evidence: Evidence


class AnnualReportRiskExtract(BaseModel):
    doc_id: str
    company_name: str = Field(description="上市公司名称")
    stock_code: Optional[str] = Field(default=None, description="证券代码")
    report_year: Literal["2024"]
    event_type: Literal["年报风险披露"]
    risk_categories: list[RiskCategory]

风险类别建议先保留较粗粒度分类，并保留原文证据；后续再做更细的行业特定风险标签。

6.5 字段抽取：规则 baseline 或硅基流动 API

字段抽取放在 MinerU 和 section checking 之后讲。原因很简单：模型或规则只能基于输入文本工作，输入 section 错了，抽取结果就会跟着错。

6.5.1 两条可选路线

路线 适合场景 优点 风险
规则 baseline 风险类别关键词较明确，如政策、竞争、汇率、原材料 透明、便宜、可复现 泛化弱，容易漏掉变体
硅基流动 API 字段表达多样、需要理解上下文 灵活，便于快速迭代 需要 API Key，必须做 JSON 校验和 evidence 检查

课堂 demo 默认先跑规则 baseline：

cd labs/cninfo_annual_report_risk_lab
python3 src/extract_fields.py --method rule
python3 src/validate_results.py

如果使用硅基流动 API，在 .env 中填写：

LLM_PROVIDER=siliconflow
LLM_BASE_URL=https://api.siliconflow.cn/v1
LLM_MODEL=Qwen/Qwen3-8B
LLM_API_KEY=your_key_here

然后运行：

python3 src/extract_fields.py --method llm
python3 src/validate_results.py

硅基流动兼容常见的 /chat/completions 调用形式。结构化抽取建议把 temperature 设为 0，并要求模型只输出合法 JSON。

6.5.2 LLM 提示词基本规则

你将收到公告中的一个目标章节。请只根据输入文本抽取字段。
如果字段不存在，输出 null。
不得根据常识补全。
每个关键字段必须给出 evidence_text。
evidence_text 必须是输入文本中的原文片段。
输出必须是合法 JSON，并符合给定字段说明。

6.5.3 常见 API 报错

  • API Key 无效；
  • model 名称错误；
  • base_url 错误；
  • 网络或超时；
  • 返回内容不是 JSON；
  • 输出太长被截断。

6.6 Section Routing 与 Section Checking

6.6.1 两者的区别

Section Routing：自动找到可能相关的章节。
Section Checking：检查找到的章节是否真的对，必要时修正规则。

Routing 是程序行为，Checking 是质量控制。一个项目可以有简单的 Routing，但不能没有 Checking。

6.6.2 为什么 section 不容易

  • 标题写法不统一；
  • 目录页会误匹配；
  • 年报章节很长；
  • 表格和段落混合；
  • 有些公告没有清晰标题；
  • 同一个关键词可能出现在“释义”“风险提示”“目录”等无关位置。

6.6.3 section_rules.yaml 示例

target_sections:
  risk_factors:
    include_keywords:
      - "风险因素"
      - "可能面对的风险"
      - "公司面临的风险"
    exclude_keywords:
      - "目录"
      - "释义"
      - "重要提示"
    min_chars: 500
    page_expansion: 1

  annual_report_risk:
    include_keywords:
      - "可能面对的风险"
      - "公司面临的风险"
      - "风险因素"
    exclude_keywords:
      - "目录"
      - "释义"
    min_chars: 500
    page_expansion: 0

6.6.4 Section 检查表

doc_id,title,target_section,found,section_title,page_start,page_end,quality_issue,notes

quality_issue 可以使用：

  • ok
  • not_found
  • toc_mismatch
  • too_short
  • table_broken
  • wrong_section
  • page_missing
  • needs_manual_review

6.7 13.5 章节级抽取与证据回填

不要把整篇年报直接交给规则或 LLM 抽取所有字段。应先定位目标章节，再把相关文本送入抽取器。

每条结果至少包含：

  • value；
  • evidence_text；
  • page_no；
  • confidence。

使用 LLM 路线时的提示词模板：

你将收到公告中的一个目标章节。请只根据输入文本抽取字段。
如果字段不存在，输出 null。
不得根据常识补全。
每个字段必须给出 evidence_text。
evidence_text 必须是输入文本中的原文片段。
输出必须是合法 JSON，并符合给定字段说明。

6.7.1 Baseline 抽取流程

  1. 随机选取 5–10 份已用 MinerU 解析的 PDF；
  2. 用 section rules 找目标章节；
  3. 人工检查 section 是否正确；
  4. 对正确 section 先跑规则 baseline，再选择是否调用硅基流动 LLM 抽取；
  5. 用 Pydantic 校验；
  6. 把校验失败写入 validation_errors.jsonl；
  7. 人工检查 evidence 是否来自输入。

6.8 课堂练习

  1. 创建 .env.example，填入 MINERU_API_KEY。
  2. 用 MinerU 解析 3–5 份 PDF，并填写 parse_check.md。
  3. 写第一版 section_rules.yaml，生成 section_check_report.csv。
  4. 为本组选题写 src/schemas.py。
  5. 先抽取 5 条规则 baseline，记录 Pydantic 错误。
  6. 如果选择 LLM 路线，再填入硅基流动 LLM_BASE_URL、LLM_MODEL、LLM_API_KEY，跑通 5 条 LLM 抽取。

6.9 Week 13 常见环境问题

问题 常见表现 建议处理
.env 没读到 提示 missing API key 确认 .env 在项目根目录，变量名和 workflow.yaml 一致
base_url 错误 404、connection error 检查是否以 /v1 结尾，脚本会自动拼接 /chat/completions
model 名称错误 provider 返回 model not found 到所用平台复制模型名称，不要凭记忆手写
JSON 不合法 Pydantic 无法解析 保存原始输出，用 JSON 修复 prompt 或降低 temperature
PDF 无法解析 parsed text 为空或乱码 先换 3–5 份样本测试，保留失败记录，必要时标记人工复核
section 命中目录 evidence 来自目录页 增加 exclude keywords、最小长度和人工 section check

6.11 常见错误

  • API Key 写进代码；
  • Prompt 没有 Schema；
  • 模型输出 JSON 但无法被 Pydantic 解析；
  • MinerU 解析结果没有检查；
  • section 命中了目录而不是正文；
  • evidence_text 是模型编造的，不在输入中；
  • page_no 缺失或错误；
  • 把整篇长 PDF 一次性塞给模型；
  • 没有保存 validation error。

6.12 教师提示

本周教师可以抽查一条结果，要求学生现场指出：

  1. 原 PDF 文件；
  2. MinerU 解析片段；
  3. section 检查记录；
  4. LLM 输出；
  5. Pydantic 校验结果；
  6. evidence_text 在原文中的位置。

如果这条链断了，说明系统还不能被信任。
